Unattended-upgrades: автоматические обновления безопасности

«Регулярно обновлять затвор — скучно. Проснуться от того, что замок проржавел и дверь открыта — ещё скучнее. — Воланд»

Введение

Настоящее руководство описывает установку и настройку пакета unattended-upgrades — механизма автоматической установки критических обновлений безопасности для Ubuntu.

Мы начинаем с сервера, на котором уже выполнены предыдущие шаги: настроен UFW (шаг 04) и Fail2ban (шаг 05). Система защищена от несанкционированного доступа. Теперь необходимо обеспечить, чтобы она оставалась защищённой и в будущем — без вашего ежедневного участия.

Цель данного руководства — настроить автоматическое обновление пакетов безопасности, исключив риск пропустить критическое исправление уязвимости из-за человеческой забывчивости.

Входные данные: вы подключены к серверу по SSH пользователем uservpn на порт 22022. Команды с sudo не запрашивают пароль. На сервере работает почтовый релей Postfix (шаг 07) — уведомления будут приходить на email.

Раздел 1. Что такое unattended-upgrades и зачем это нужно

1.1. Краткая теория

Unattended-upgrades — пакет, который автоматически устанавливает обновления пакетов в фоновом режиме. Он запускается ежедневно через systemd-таймер и обрабатывает очереди обновлений, не требуя вмешательства администратора.

Без него вы обязаны регулярно заходить на сервер и выполнять apt update && apt upgrade. Если вы когда-нибудь пропустите месяц-другой — сервер будет работать с известными уязвимостями. С unattended-upgrades критические исправления CVE ставятся в течение суток после выхода.

1.2. Риски

Риск Как unattended-upgrades его адресует
Пропустили обновление безопасности Установит автоматически, письмо на почту
Обновление сломало сервис Настраивается только security-репозиторий (минимальный риск)
Закончилось место на диске AutocleanInterval удаляет устаревшие пакеты
Требуется перезагрузка после обновления ядра Отправляет письмо, но НЕ перезагружает сервер

1.3. Что будет настроено

Параметр Значение Пояснение
Частота проверки обновлений Ежедневно APT::Periodic::Update-Package-Lists "1"
Частота установки обновлений Ежедневно APT::Periodic::Unattended-Upgrade "1"
Очистка кэша apt Раз в 7 дней AutocleanInterval "7"
Источники обновлений Только security ${distro_id}:${distro_codename}-security
Email для отчётов sergey@poleckiy.ru Письмо только при изменениях
Автоматическая перезагрузка Отключена Вы решаете, когда перезагружать

Раздел 2. Установка unattended-upgrades

2.1. Проверка наличия

В Ubuntu unattended-upgrades часто предустановлен. Проверим:

dpkg -l unattended-upgrades | grep ^ii

Если видите строку с ii — пакет установлен. Если нет — устанавливаем.

2.2. Установка (если не установлен)

sudo apt update
sudo apt install -y unattended-upgrades

Раздел 3. Настройка

3.1. Включить автоматическое обновление

Создайте или отредактируйте /etc/apt/apt.conf.d/20auto-upgrades:

sudo tee /etc/apt/apt.conf.d/20auto-upgrades << 'EOF'
APT::Periodic::Update-Package-Lists "1";
APT::Periodic::Unattended-Upgrade "1";
APT::Periodic::AutocleanInterval "7";
APT::Periodic::Unattended-Upgrade-Mail "sergey@poleckiy.ru";
EOF
Параметр Значение Что делает
Update-Package-Lists "1" 1 (раз в день) Обновляет список пакетов (apt update)
Unattended-Upgrade "1" 1 (раз в день) Устанавливает обновления
AutocleanInterval "7" 7 (раз в неделю) Чистит кэш apt от старых версий
Unattended-Upgrade-Mail email Кому слать отчёт об обновлениях

3.2. Настроить политику обновлений

Отредактируйте /etc/apt/apt.conf.d/50unattended-upgrades:

sudo tee /etc/apt/apt.conf.d/50unattended-upgrades << 'EOF'
// Обновлять только security-пакеты
Unattended-Upgrade::Allowed-Origins {
    "${distro_id}:${distro_codename}-security";
};

// Письмо с отчётом администратору
Unattended-Upgrade::Mail "sergey@poleckiy.ru";

// НЕ перезагружать автоматически
Unattended-Upgrade::Automatic-Reboot "false";

// Письмо если требуется перезагрузка (kernel обновлён)
Unattended-Upgrade::MailReport "on-change";

// Удалять неиспользуемые зависимости
Unattended-Upgrade::Remove-Unused-Dependencies "true";

// Удалять неиспользуемые kernel-пакеты
Unattended-Upgrade::Remove-Unused-Kernel-Packages "true";
EOF

3.3. Ключевые параметры

Allowed-Origins — самый важный параметр. Мы ограничиваем установку только пакетами из репозитория -security. Это исключает риск того, что автоматическое обновление установит новую версию PHP или Nginx с изменениями API, которые сломают сайт.

Если вы хотите получать все обновления (включая обычные -updates), можно добавить:

"${distro_id}:${distro_codename}-updates";

Но на production-сервере это не рекомендуется — обычные обновления могут содержать изменения поведения.

MailReport "on-change" — письмо приходит только когда unattended-upgrades реально что-то установил. Если всё актуально — тишина. Это осмысленное поведение: если письмо не пришло — значит сервер актуален.

«Зачем беспокоить гражданина по пустякам? — пожал плечами Коровьев. — Если ничего не случилось — значит, всё в порядке. Если случилось — вы узнаете. Телодвижения излишни.»

Раздел 4. Проверка работы

4.1. Сухой прогон

Проверьте, какие обновления будут установлены, без фактической установки:

sudo unattended-upgrades --dry-run --debug

Эта команда покажет:

  • Какие пакеты доступны для обновления
  • Будут ли они установлены (согласно политике Allowed-Origins)
  • Любые ошибки конфигурации

4.2. Проверка статуса службы

systemctl status unattended-upgrades

Должен быть active (running).

4.3. Просмотр лога

sudo journalctl -u unattended-upgrades --since "1 week ago" --no-pager

Лог показывает, какие пакеты были обновлены и когда.

4.4. Имитация отчёта

Запросите тестовый отчёт:

sudo unattended-upgrades --send-mail --dry-run

Письмо с тестовым отчётом должно прийти на указанный email.

Раздел 5. Диагностика

Письма не приходят

# Проверить почтовый лог
sudo tail -50 /var/log/mail.log

# Проверить, что Postfix работает
systemctl status postfix

# Отправить тестовое письмо вручную
echo "Test" | sudo mail -s "Test from unattended-upgrades" sergey@poleckiy.ru

Обновления не устанавливаются

# Проверить конфигурацию
sudo unattended-upgrades --dry-run --debug 2>&1 | grep -i "allowed\|origin\|package"

# Проверить доступность репозиториев
sudo apt update

Раздел 6. Важные замечания

6.1. Перезагрузка после обновления ядра

Unattended-upgrades может обновить ядро Linux, но не перезагрузит сервер (мы это явно запретили параметром Automatic-Reboot "false"). После обновления ядра вы получите письмо, и вам нужно будет перезагрузить сервер в удобное время:

sudo reboot

Проверить, что новое ядро загружено:

uname -r

Сравните с установленными версиями:

dpkg -l | grep linux-image

6.2. Мониторинг

Рекомендуется раз в месяц проверять, что unattended-upgrades работает корректно:

sudo unattended-upgrades --dry-run --debug 2>&1 | tail -20

«Вот и всё, — подытожил Воланд. — Система будет обновляться сама, письма — приходить сами, а вы — заниматься тем, чем считаете нужным. Не благодарите. Порядок, как известно, должен быть в первую очередь в голове, а уж потом — на сервере.»
