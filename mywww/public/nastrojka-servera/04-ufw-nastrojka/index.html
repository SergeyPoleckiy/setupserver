UFW: настройка фаервола

«Запереть дверь — ещё не значит обезопасить дом. Но оставить её открытой — значит, гарантированно потерять всё, что внутри. — Воланд»

Введение

Настоящее руководство описывает установку и настройку фаервола UFW (Uncomplicated Firewall) — удобного фронтенда для управления правилами сетевой фильтрации (iptables/nftables) в Ubuntu.

Мы начинаем с сервера, на котором выполнены предыдущие шаги: SSH настроен (только по ключам, вход по паролям запрещён), пользователь user01 имеет права sudo без пароля, и базовая система полностью подготовлена.

Цель данного руководства — закрыть все входящие сетевые порты, кроме тех, которые действительно необходимы для работы сервисов, и настроить ограничение частоты подключений (rate limiting) для SSH.

Все шаги пронумерованы и должны выполняться в указанном порядке.

Входные данные: вы подключены к серверу srv02 по SSH пользователем user01 на стандартный порт 22. Команды с sudo не запрашивают пароль.

Раздел 1. Зачем нужен фаервол

1.1. Краткая теория

Фаервол (межсетевой экран) фильтрует входящий и исходящий трафик. Он решает, какие пакеты пропустить в систему, какие заблокировать, а какие ограничить.

Без фаервола все сетевые порты сервера открыты миру. Любой бот или сканер уязвимостей может простукивать запущенные у вас сервисы. Это равносильно открытой двери в квартиру в неблагополучном районе города.

1.2. Почему именно UFW

UFW — это стандартное и крайне простое средство для фильтрации трафика в Ubuntu. Вместо громоздких правил iptables вы используете человекочитаемые команды вида ufw allow 22/tcp.

1.3. Какие порты открываем сейчас

На этапе базовой настройки сервера открыт только один порт:

Порт Протокол Правило Сервис Назначение
22 TCP LIMIT SSH Удалённый доступ (ограничение частоты для защиты от брутфорса)

Все остальные входящие порты заблокированы. По мере установки служб (веб-сервер, VPN, CMS) мы будем открывать соответствующие порты — это отражено в последующих руководствах.

Исходящий трафик разрешён без ограничений (для скачивания обновлений, отправки логов и т.д.).

Коровьев, разглядывая таблицу портов: «Одна открытая дверь из десятков тысяч. Остальные — наглухо заперты. Со временем мы откроем ещё, но ровно столько, сколько потребуется. Никакой самодеятельности — только по делу.»

Раздел 2. Установка UFW

2.1. Проверка наличия пакета

В большинстве образов Ubuntu 24.04 UFW уже предустановлен. Проверим версию:

sudo ufw version

Если утилита не найдена, установите её:

sudo apt update
sudo apt install -y ufw

2.2. Проверка статуса

Сразу после установки UFW находится в неактивном состоянии:

sudo ufw status

Ожидаемый вывод:

Status: inactive

Важно: Ни в коем случае не включайте фаервол до добавления разрешающих правил! Если сделать это, вы мгновенно разорвёте текущую сессию SSH и заблокируете себе доступ к серверу.

Раздел 3. Настройка политик по умолчанию

3.1. Принцип «Белого списка»

Мы настроим политики по умолчанию: запретим все входящие запросы и разрешим все исходящие.

sudo ufw default deny incoming
sudo ufw default allow outgoing

Ожидаемый вывод:

Default incoming policy changed to 'deny'
(be sure to update your rules accordingly)

Воланд кивнул: «Теперь сервер по умолчанию глух и нем для внешнего мира. Всё, что мы разрешим — будет сделано осознанно. Это называется принцип наименьших привилегий, и он спас больше систем, чем все антивирусы вместе взятые.»

Раздел 4. Настройка логирования

UFW позволяет записывать информацию о заблокированных пакетах в лог-файлы. Включим умеренный уровень логирования, чтобы не перегружать накопитель лишней информацией:

sudo ufw logging low

Раздел 5. Добавление правил доступа

5.1. SSH (порт 22) с ограничением частоты (Limit)

Вместо обычного allow мы применим директиву limit. Она разрешает доступ, но блокирует IP-адрес атакующего, если с него зафиксировано более 6 новых подключений в течение 30 секунд. Это отличная базовая защита от подбора ключей и флуда:

sudo ufw limit 22/tcp

Примечание: В последующих руководствах, по мере установки веб-сервера, VPN и CMS, мы будем возвращаться к UFW и добавлять правила для портов 80, 443, 8443, 4443, 2096. Сейчас открыт только порт для SSH, и это правильно — минимум поверхности для атаки.

Раздел 6. Активация фаервола

6.1. Предварительная проверка правил

Перед тем как запустить фаервол, выведите список подготовленных правил:

sudo ufw show added

Ожидаемый вывод:

Added rules:
ufw limit 22/tcp

Внимание: В списке обязательно должно быть правило limit 22/tcp (или allow 22/tcp). Если его там нет — остановитесь, не активируйте UFW и добавьте правило заново.

6.2. Включение UFW

Включите фаервол:

sudo ufw --force enable

Флаг --force отключает интерактивное предупреждение о возможном разрыве сессии SSH. Так как правило для SSH добавлено, соединение не прервётся.

Убедитесь, что статус изменился на active:

sudo ufw status verbose

Вы увидите активный фаервол, политики deny (incoming), allow (outgoing) и полный список задействованных правил (включая автоматические правила для протокола IPv6).

Раздел 7. Тестирование

7.1. Проверка SSH

Не закрывая текущий терминал, откройте новое окно на своём компьютере и подключитесь к серверу по SSH:

ssh user01@<IP-адрес-сервера>

Если вы успешно вошли в систему, значит, фаервол функционирует корректно и не блокирует администратора.

7.2. Проверка блокировки

Попробуйте проверить доступность закрытого порта (например, порта базы данных MySQL 3306) с внешнего компьютера:

nc -zv <IP-адрес-сервера> 3306

Соединение должно завершиться ошибкой по таймауту, поскольку UFW молча отбрасывает (DROP) такие пакеты. Попытка подключения отразится в журнале событий. Посмотреть заблокированные пакеты можно командой:

sudo dmesg | grep -i 'UFW BLOCK' | tail -5

Воланд, просматривая логи: «Вот видите — уже кто-то стучится в закрытый порт. А MySQL у нас даже не установлен. Фаервол вежливо промолчал, и записал это в журнал — для потомков. Через неделю таких записей будут тысячи. Это нормально. Интернет — место шумное.»

Раздел 8. Управление правилами в будущем

Если вам потребуется удалить правило или изменить его:

  1. Выведите список правил с номерами: 
    sudo ufw status numbered
  2. Удалите ненужное правило по его индексу: 
    sudo ufw delete <номер_правила>

Раздел 9. Итоги

На данном этапе мы имеем:

Параметр Статус
UFW установлен и запущен ✅
Политика по умолчанию: deny incoming ✅
Политика по умолчанию: allow outgoing ✅
Включён rate-limiting на порту SSH (22) ✅
Логирование заблокированных пакетов запущено ✅

Конец документа. Настроившему — спокойный сон, забывшему правило SSH — да поможет ufw reset и веб-консоль провайдера.
